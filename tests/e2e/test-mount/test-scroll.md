# 滚动测试文档

这个文档用于测试光标和滚动位置的保存与恢复。需要足够长的内容才能产生滚动。

## 第一节：基础概念

在软件开发中，光标位置和滚动位置的恢复是一个重要的用户体验功能。当用户刷新页面或者重新打开应用时，能够回到之前的位置可以大大提高工作效率。

本文档的第一部分介绍了为什么需要光标恢复功能。当我们在长文档中编辑时，经常会因为各种原因需要刷新页面。如果没有光标恢复功能，每次刷新后都需要手动找到之前编辑的位置，这非常浪费时间。

### 1.1 问题描述

在传统的Web应用中，页面刷新会导致所有客户端状态丢失。这包括：
1. 光标在文档中的位置
2. 滚动条的位置
3. 用户正在进行的编辑操作
4. 会话历史记录

对于短文档来说，这不是一个大问题。但对于长文档（如技术文档、论文、书籍章节），重新定位到之前的编辑位置可能会非常耗时。

### 1.2 解决方案概述

我们的解决方案是将光标和滚动信息保存到浏览器的 localStorage 中。具体实现包括：

- 在切换文件前保存当前位置
- 在页面卸载前保存当前位置
- 在页面加载后恢复上次保存的位置
- 使用 headingText + scrollPercent 的方式定位

## 第二节：技术实现细节

### 2.1 保存机制

保存机制主要涉及以下几个方面：

首先，我们需要捕获光标的当前位置。对于 IR 模式（即时渲染模式），我们通过 TreeWalker 遍历文本节点来计算光标的文本偏移量。对于 SV 模式（分屏模式），我们直接获取 textarea 的 selectionStart。

其次，我们需要计算滚动位置。我们使用 scroll百分比而不是绝对像素值，这样即使文档内容发生变化（如窗口大小改变），也能保持相对位置的一致性。

最后，我们还需要记录光标附近的标题文本。这用于在刷新后快速定位到正确的标题位置。

### 2.2 恢复机制

恢复机制需要考虑多种情况：

1. 正常刷新页面后的恢复
2. 从其他文件切换回来后的恢复
3. 浏览器崩溃后的恢复

对于每种情况，我们都使用相同的核心逻辑：从 localStorage 读取保存的位置信息，然后在 Vditor 渲染完成后应用这些信息。

### 2.3 Vditor 渲染时序

Vditor 的渲染是异步的。调用 `setValue()` 后，DOM 的更新需要一定时间。因此我们在 `after` 回调中使用多个 setTimeout 来确保恢复时机正确。

具体延时设置为 100ms、400ms 和 800ms 三个时间点。第一个时间点处理快速渲染的情况，第二个处理正常情况，第三个作为兜底。

## 第三节：测试策略

### 3.1 单元测试

单元测试主要验证保存和恢复函数的正确性。包括：
- 光标位置计算的准确性
- 滚动百分比计算的准确性
- heading 查找的正确性

### 3.2 端到端测试

端到端测试模拟真实用户的操作场景：
- 打开文件 → 滚动到指定位置 → 刷新页面 → 验证位置恢复
- 打开文件 → 点击大纲标题 → 刷新页面 → 验证标题恢复
- 切换文件 → 验证 localStorage 中保存了位置信息

### 3.3 边界条件测试

边界条件包括：
- 空文档（无内容可滚动）
- 超长文档（需要大量滚动）
- 没有标题的文档
- 只有标题没有内容的文档

## 第四节：性能考虑

### 4.1 localStorage 读写性能

localStorage 的读写是同步的，但在大量数据时仍可能影响性能。我们的策略是：
- 只在必要时保存（页面卸载、文件切换）
- 保存的数据量尽量小（只保存必要的位置信息）
- 使用 JSON.stringify/parse 进行序列化

### 4.2 Vditor 渲染性能

恢复位置时，Vditor 可能需要重新渲染部分 DOM。我们通过以下方式减少性能影响：
- 使用 requestAnimationFrame 避免阻塞主线程
- 批量处理 DOM 操作
- 避免不必要的重排和重绘

### 4.3 内存使用

保存的位置信息只包含几个数字和字符串，内存占用可以忽略不计。但我们仍需注意：
- 及时清理过期的位置信息
- 避免在内存中缓存大量历史位置
- 使用弱引用（如可能）

## 第五节：总结

光标和滚动位置的恢复是一个看似简单但实际需要仔细处理的功能。它涉及浏览器的各种 API、Vditor 的内部实现、以及用户体验的多个方面。

通过 localStorage 保存位置信息，结合 Vditor 的渲染时序控制，我们实现了一个可靠的位置恢复功能。这不仅提高了用户的工作效率，也增强了应用的整体体验。

未来可能的改进包括：
1. 支持更多的编辑器模式
2. 添加位置历史的版本管理
3. 支持跨设备的同步恢复
4. 提供更精确的光标恢复算法
